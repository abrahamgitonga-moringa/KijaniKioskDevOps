## Question 1: The State File as a System

### How Terraform Learns Extended Attributes
Terraform discovers and records un-declared attributes through a **Post-Provisioning API Synchronization Callback**. When you define a resource, your HCL file only specifies the inputs (the desired constraints). 

During the `apply` phase, Terraform sends these constraints to the target provider or hypervisor API. Once the infrastructure engine allocates the resource, its backend daemon computes and assigns operational system values (e.g., specific hardware IDs, private network IP mappings, interface MAC addresses, and lease launch times). 

The provider API returns this complete payload back to Terraform as a JSON response. Terraform captures this incoming metadata and saves it directly into `terraform.tfstate`.

### The Lifecycle of State During Destructive Phases
* **When running `terraform destroy`:** Terraform does not immediately wipe the state file. Instead, it reads the existing state file to find the unique tracking IDs of the live assets. It maps these IDs to API deletion endpoints to tear down the infrastructure. Once the backend engine confirms the assets are completely deleted, Terraform scrubs the resource blocks from the state file, rendering it empty (`[]`).
* **If the state file is manually deleted without running a destroy command:** Terraform loses its memory of the managed infrastructure. On the next `terraform plan`, Terraform treats your configuration file as a brand-new deployment request. Because it has no tracking records mapping your local HCL to the existing live assets, it will generate a plan to **create completely duplicate infrastructure assets**, leaving your existing VMs running as untracked "orphaned" resources that continue to accumulate costs.

### The Correct Recovery Procedure
If a state file is lost but the infrastructure still exists, you must reconstruct it using:
1. Run `terraform init` to initialize a clean backend state schema.
2. Locate the corresponding real-world identifier from your hypervisor or cloud console (e.g., the Multipass instance name or an AWS instance ID).
3. Bind the live asset back to your code tracking framework by executing the import command:

## Question 2: The (known after apply) Values

### Why Certain Values Cannot Be Known at Plan Time
Values are marked as `(known after apply)` because they are **dynamically generated by the hosting runtime provider during actual compilation**, rather than being mathematically calculated from your input code. A `terraform plan` is merely an isolated parsing of local configuration files; it does not allocate real-world hardware or compute live cloud-fabric details.

### Specific Examples
1. **`id` (Resource Tracking Identifier):** In a resource block, the definitive tracking handle is a random token generated by the virtualization engine at the exact moment of initialization. Terraform cannot predict this randomized, unique integer before the asset is instantiated.
2. **`private_ip` / `public_ip` (Network Allocations):** While you may request a general subnet block, the specific internal IP address assignment depends on which dynamic leases are open in the hypervisor's DHCP pool at the millisecond the instance initializes.

### Dependent Outputs During Plan Time
If an explicit output block depends on an attribute flagged as `(known after apply)`, Terraform will suppress the final scalar string value during `terraform plan`. Instead, the planning summary output will explicitly display:
`api_server_ip = (known after apply)`
## Question 3: Hardcoded vs Variable

### The Shared Team Configuration Vulnerability
Hardcoding a specific workstation IP address (e.g., `197.248.42.15/32`) into an infrastructure security boundary block creates several issues for a team:
* **Deployment Blockers:** The configuration immediately fails for any other engineer (like Nia or Tendo) who pulls the repository down, because their local network gateways will be blocked by your specific firewall whitelist rule.
* **Brittle Code:** If your local ISP shifts your lease dynamically, your automation breaks until you manually change the source code file.
* **Security Leaks:** Committing hardcoded localized network parameters or sensitive home network paths directly into Git violates core security practices.

### The Solution: Variable Parameterization
We decouple this by lifting the access parameter out of the resource block and defining it as an injection variable. For a variable structured to manage an array of values, you must declare its type as a **`list(string)`**. 

```hcl
variable "allowed_admin_ips" {
  type        = list(string)
  description = "A whitelisted list of IPv4 CIDR blocks authorized to establish inbound operational sessions."
  default     = ["127.0.0.1/32"] 
}
ingress {
  from_port   = 22
  to_port     = 22
  protocol    = "tcp"
  cidr_blocks = var.allowed_admin_ips
}
## Question 4: What Tuesday's Configuration Cannot Do

### Staging vs. Production Divergence Gaps
If you attempt to run today's single-file, unparameterized configuration against a production tier, several parameters would need to be changed manually within the file:
* **Sizing/Instance Type Profiles:** Staging might run fine on low-cost compute targets (e.g., micro instances), but production demands robust configurations with higher performance limits.
* **Environment Tags:** The metadata tracking tags (`Environment = "staging"`) must be updated to ensure clear billing and logging separation.
* **Target Environment Networks:** The underlying tracking IP or destination VPC subnet routing spaces must be changed to prevent production traffic from routing into staging workloads.

### The Breakdown of Today's Single-File Approach
If production demands a larger footprint, a separate VPC network, and twelve scaled VMs instead of one, handling this using today's single-file approach forces you into the **"Copy-and-Paste" Anti-Pattern**. You duplicate your entire codebase into a separate folder named `/production` and modify the values manually. 

**What Breaks:** This creates **Configuration Drift**. If Tendo updates a security firewall rule or changes an application deployment parameter in the staging directory next Monday, someone has to manually remember to copy that exact structural change into the production directory. If they forget, production slips out of sync, defeating the purpose of Infrastructure as Code. Wednesday introduces modules, backends, and multi-variable inputs to transform this configuration into a dynamic template that can stamp out isolated environments safely from a single source of truth.
